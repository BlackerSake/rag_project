## 二、压缩触发记录

### 2.1 判断机制

```
每轮对话结束后执行一次判断：
  tokens_used  >  threshold(5734)  →  should = True  →  触发压缩
  tokens_used  ≤  threshold(5734)  →  should = False →  跳过
```

> [!tip] 关键设计点
> - 阈值固定为 `5734`== 0.7 * max_token，但**触发时机由实际 token 消耗决定**，不是每轮必压缩
> - `rounds=2` 单轮对话内可以连续触发多次压缩
> - 随着对话积累，summary 本身也在增长（542 → 1461），但始终比原文 messages 小得多

## 三、一次压缩的完整链路

> 以 `21:52:04` 这次为例（tokens=11219，压缩前 7 条消息）

```log
# ① 判断
21:52:04,222  core.summary.compression  INFO
  上下文压缩判断: should=True tokens=11219 threshold=5734 messages=7 tool_tokens=0 rounds=1

# ② 触发
21:52:04,222  core.edges  INFO  上下文达到压缩条件，触发压缩
21:52:04,225  core.nodes  INFO  进入上下文压缩节点

# ③ 调用 LLM 生成 rolling summary（耗时 ~8s）
21:52:04,581  httpx  INFO  POST https://api.deepseek.com/chat/completions "200 OK"
21:52:12,045  core.summary.summarizer  INFO  滚动摘要生成完成，长度=904

# ④ 写回
21:52:12,046  core.summary.workflow  INFO
  上下文压缩完成: old_messages=2 recent_messages=5 tool_results=0
```

### 压缩前后 messages 变化

```
压缩前：  [ msg1 | msg2 | msg3 | msg4 | msg5 | msg6 | msg7 ]   → 7 条原文
                                                                          ↓
压缩后：  [ summary(msg1+msg2) | msg3 | msg4 | msg5 | msg6 | msg7 ]
           ↑ 旧2条合并为摘要      ↑ 最近5条原文保留不动
```

| 字段 | 值 | 含义 |
|---|---|---|
| `old_messages` | 2 | 最旧的2条被压缩进 summary |
| `recent_messages` | 5 | 最近5条原文保留 |
| `tool_results` | 0 | 本次无工具结果被压缩 |

## 六、Rolling Summary 的固定结构

每次压缩生成的 summary 均遵循以下 7 个字段：

```
[当前目标]        ← 用户此轮的主要查询意图（含完成/进行/待确认状态）
[用户偏好与约束]  ← 用户习惯和要求，包括负面体验（如"极度反感闲聊"）
[已确认事实]      ← 业务数据（单号/状态/金额）+ 用户确认的非业务事实
[重要决策]        ← 用户做出的明确选择（放弃某查询、优先处理某问题）
[工具最终结果]    ← 本轮工具调用的返回值（物流查询结果等）
[未解决问题]      ← 尚未处理完的事项，下一轮继续跟进
[下一步计划]      ← 助手当前规划的下一步动作
[已废弃上下文]    ← 不再适用的旧信息 + 废弃原因（避免旧数据干扰推理）
```

> [!tip] 废弃上下文的价值
> `[已废弃上下文]` 字段不是简单删除，而是**记录「曾经有过但现在无效」**。
> 例如：用户曾说放弃某单号查询 → 后来否认 → summary 把该条目从废弃中恢复，并标注「用户否认，恢复为待处理」。

## 七、边界说明

> [!success] 日志可以证明
> - 基于 token 数**动态判断**是否触发压缩（不是每轮固定压缩）
> - 触发后调用 LLM 生成 rolling summary
> - 压缩后**保留最近 5 条原文消息**
> - 后续节点能**读取 summary 并正常使用**
> - summary 同时保存**业务事实和非业务上下文**（纠错、情绪）
> - 冲突解决状态、废弃上下文均被跟踪

> [!failure] 日志不能证明
> - 跨进程/重启后的持久化能力
> - 跨用户的数据隔离机制
> - 跨设备的长期记忆同步
> - 向量化语义检索能力

> 该模块的准确定位是：**会话级 rolling summary 上下文压缩模块**，而非完整的用户长期记忆系统。
