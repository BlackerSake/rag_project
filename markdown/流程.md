逻辑链路与语言版本的综合修改方向。
knowledge_base.py 是主战场，绝大多数修改集中在这里——连接探活、重连逻辑、fast-path 锁、_vector_search 自愈、_generate_doc_id 的哈希替换、模块级副作用清理，这些都是 KnowledgeBase 类自身的问题。
init.py 只需要做一处联动修改：把 check_knowledge_base() 里直接操作 MILVUS_CHECK_ALIAS 的那段 pymilvus 裸调用，替换成委托给 knowledge_base.get_milvus_collection_stats() 方法，同时删掉 MILVUS_CHECK_ALIAS 常量。这样两套连接别名的双轨问题就消除了。
其余 init.py 的结构——retry()、FAQRepository、KnowledgeBaseInitializer、主流程编排——全部不动，因为它们本身设计是合理的，问题不在那里。
简单说就是：knowledge_base.py 改连接管理核心逻辑，init.py 只做一处对齐性修改来消除双轨冲突。

## 一、逻辑链路修改方向

### 当前链路（有缺陷）

```
search() 调用
    │
    ▼
ensure_connected()
    │
    ├─ 外层 if: vector_store is None OR es_store is None
    │       每次 search 都检查，高并发下每次都 acquire _init_lock ← 热点锁
    │
    └─ 锁内: has_connection("default")
                │
                ├─ True  → 直接信任，跳过探活 ← gRPC channel 可能已死
                │           logger.info("已连接，无需重复连接")
                │
                └─ False → connections.connect(...)
                                │
                                └─ 失败 → return None ← 静默失败
                                                         vector_store 维持 None
                                                         下次 search 重复此流程

_vector_search() 调用
    │
    └─ 失败 → logger.error() → return [] ← 不重连，不自愈
                                            每次 search 都失败直到进程重启

check_knowledge_base() [init.py]
    │
    └─ 直接操作 MILVUS_CHECK_ALIAS ← 与 KnowledgeBase 的 "default" 双轨并存
                                       互不感知，存在互相 remove 的风险
```

### 建议链路（修改后）

```
search() 调用
    │
    ▼
ensure_connected()
    │
    ├─ fast-path: _initialized.is_set() → True → 直接返回，零锁开销 ← 新增 Event
    │
    └─ 首次/重置时 acquire _init_lock（二次检查防重入）
            │
            ▼
        _is_milvus_connection_alive()   ← 新增：真实 RPC 探活
            │
            ├─ 存活 → 直接 _init_milvus_vector_store()
            │
            └─ 不存活 → _connect_pymilvus()  ← 新增：指数退避重连
                            │
                            ├─ 成功 → _init_milvus_vector_store()
                            └─ 失败 → 记录日志，继续尝试 ES
            │
            └─ _initialized.set()  ← 标记完成，后续 fast-path 生效

_vector_search() 调用
    │
    └─ 失败
          │
          ├─ _is_connection_error() == True  ← 新增：识别连接层异常
          │       │
          │       └─ vector_store = None → ensure_connected() → 重试一次
          │               │
          │               ├─ 成功 → 返回结果
          │               └─ 失败 → 降级到 BM25 / 兜底
          │
          └─ _is_connection_error() == False → 业务异常，直接降级

check_knowledge_base() [init.py]
    │
    └─ 委托给 knowledge_base.get_milvus_collection_stats()  ← 消除双轨
            │
            └─ 复用 "default" 连接，通过封装层查询，不直接操作 pymilvus
```

## 二、Python 语言层面的修改方向

### 2.1 `hash()` → `hashlib.sha256`（跨进程确定性）

```python
# ❌ 现状：PYTHONHASHSEED 随机化导致每次进程启动 hash 值不同
# _merge_results_with_rrf 依赖 doc_id 去重，跨请求结果不稳定
content_hash = hash(doc.page_content + json.dumps(doc.metadata, sort_keys=True))
return f"{uuid.uuid4().hex[:8]}_{content_hash}"
#        ↑ uuid4 随机前缀 + hash 随机值 = 完全不可重复

# ✅ 建议：SHA-256 截断，相同内容 → 相同 ID，跨进程稳定
import hashlib
payload = doc.page_content + json.dumps(doc.metadata, sort_keys=True, ensure_ascii=False)
return hashlib.sha256(payload.encode("utf-8")).hexdigest()[:32]
```

### 2.2 裸 `except:` → `except Exception:`（不吞 BaseException）

```python
# ❌ 现状：吞掉 KeyboardInterrupt / SystemExit，进程无法正常退出
try:
    utility.has_collection("dummy", using=self.alias)
    return True
except:          # ← 捕获一切，包括系统信号
    return False

# ✅ 建议
try:
    utility.list_collections(using=alias, timeout=3)
    return True
except Exception:   # ← 只捕获业务/网络异常
    return False
```

### 2.3 模块级副作用 → 延迟到类内（`load_dotenv` 与 `logging.basicConfig`）

```python
# ❌ 现状：import knowledge_base 时立即执行以下副作用
# 1. 读写文件系统（load_dotenv）
# 2. 修改全局 logging 配置（basicConfig）
# 3. 读取全局环境变量（dashscope_api_key 等模块级变量）
# 当 init.py 也调用 configure_logging() 时，两处 basicConfig 互相覆盖

load_dotenv(dotenv_path=env_path)                    # ← 模块级副作用
logging.basicConfig(...)                              # ← 覆盖 init.py 的日志配置
dashscope_api_key = os.getenv("DASHSCOPE_API_KEY")   # ← import 时固化，.env 未必已加载

# ✅ 建议：将环境变量读取移入 __init__，logging 由调用方统一配置
class KnowledgeBase:
    def __init__(self):
        # 延迟读取，确保 .env 已由调用方（init.py）加载完毕
        self._dashscope_api_key = os.getenv("DASHSCOPE_API_KEY")
        self._dashscope_model_id = os.getenv("DASHSCOPE_MODEL_ID")
        self._milvus_uri = self._normalize_uri(
            os.getenv("MILVUS_URI", "http://localhost:19530")
        )
        # basicConfig 完全移除，logger 只做 getLogger
```

### 2.4 `threading.Event` 替代 `is None` 重复检查（fast-path 无锁）

```python
# ❌ 现状：每次 search → ensure_connected → 都要检查两个 is None
# 高并发时每次都 acquire _init_lock，即便已初始化完毕
if self.vector_store is None or self.elasticsearch_store is None:
    with self._init_lock:
        ...

# ✅ 建议：Event.is_set() 是原子操作，无锁，fast-path 开销接近零
# __init__ 中：
self._initialized = threading.Event()

# ensure_connected 中：
if self._initialized.is_set():       # fast-path，绝大多数请求走这里
    return self.vector_store
with self._init_lock:
    if self._initialized.is_set():   # 二次检查，防止并发初始化
        return self.vector_store
    # ... 执行初始化 ...
    self._initialized.set()
```

### 2.5 固定重试间隔 → 指数退避（连接重试策略）

```python
# ❌ 现状：固定 2s 间隔，不适合 Milvus 节点重启（需要更长等待）
# 也不适合网络瞬断（2s 过长）
time.sleep(2)

# ✅ 建议：指数退避 + 上限封顶
for attempt in range(max_retries):
    ...
    wait = min(2 ** attempt, 30)   # 1s, 2s, 4s, ..., 最大 30s
    time.sleep(wait)
```

### 2.6 `f-string` 日志 → `%` 占位符（与 `init.py` 风格统一）

```python
# ❌ 现状：knowledge_base.py 全部使用 f-string 日志
# init.py 全部使用 % 占位符（lazy evaluation，logger 未启用时不格式化字符串）
logger.error(f"向量检索失败: {str(e)}")         # ← 无论日志级别都执行格式化

# ✅ 建议：统一用 % 占位符，与 init.py 风格一致，性能更优
logger.error("向量检索失败: %s", e)             # ← 日志级别不满足时跳过格式化
logger.info("向量检索成功，返回 %s 条结果", len(vector_results))
```

### 2.7 `MilvusClient` 无效导入清理

```python
# ❌ 提案中导入了 MilvusClient 但从未使用
# MilvusClient 和 connections 是 pymilvus 两套独立 API，混用导致连接池冲突
from pymilvus import MilvusClient, connections, Collection   # MilvusClient 无用
import functools                                              # functools 无用

# ✅ 只保留实际使用的符号
from pymilvus import connections, utility, Collection
```

## 三、修改优先级汇总

```
P0（直接影响稳定性，必须修复）
├─ 2.1  hash() → hashlib.sha256          RRF 去重逻辑依赖此正确性
├─ 链路  _is_milvus_connection_alive()   根治 "should create connection first"
└─ 链路  _vector_search 一次性重连恢复   线上自愈能力

P1（影响并发性能与正确性）
├─ 2.4  threading.Event fast-path        消除高并发锁争用
├─ 2.3  模块级副作用清理                 消除与 init.py 的日志配置冲突
└─ 链路  check_knowledge_base 消除双轨   消除两套别名互相干扰的风险

P2（代码质量与规范性）
├─ 2.2  裸 except → except Exception
├─ 2.5  固定间隔 → 指数退避
├─ 2.6  f-string → % 占位符日志
└─ 2.7  清理无效 import
```
