### 方案一：Query 改写与拆解（解决“多问”）
既然单一的意图匹配只能给出一个 intent_id，那我们就先让 LLM 把这一句长话拆开。这是目前 Agent 架构中处理复杂逻辑的标准做法。

逻辑流：
分析节点 (Decomposer Node)：
LLM 接收原始问题，输出一个 JSON 列表。
输入：“我想查询一下我的快递到了哪里了。还没送到。如果明天还不到我就退款。”
输出：["我想查询快递物流状态", "我想了解退款政策/申请退款"]
循环/并行检索：对列表中的每一个子问题分别运行 match_intent 和 retrieve_knowledge。
聚合回答：将搜到的 A3 和 A1 的知识点同时喂给 direct_answer 节点。

### 进一步修改:
不再要求 LLM 输出 JSON 数组，而是利用 scopeEmbedding 对整句话进行向量化，并在 Milvus 中“多捞鱼”，通过结果反推子问题。
